Props

Since we've begun talking about them, their usage begs the question: what exactly are props? A simple definition: props are read-only properties passed down from a parent component to child, which begs a similar question. The answer:

// Sample component.
// Notice that props is just an object passed as a function argument
const child = props => <h1>{props.foo}</h1>;

<Parent>
  <Child className="my-class" foo="Foo!"/>
  <AnotherChild bar="42" style={/* styles are props */}>
    <GrandChild baz="Props can be called anything we like!" />
  <AnotherChild>
</Parent>

In short: props are anything we want to pass to a React component—this is easy to think about as just passing arguments to a function, and we can conceptualize React components as JavaScript functions. Components exist to describe and return some part of our application's UI; props are read-only arguments to them.

We can demo this easily in a sandbox React app. Note that since we can provide any props we'd like, we can reuse these components in different situations. Props can contain any datatype and other components—this is useful for "wrapper" elements and higher-order-components, the latter of which we'll defer on explaining until later.

In passing props from parent to child, we do so in a top-down, unidirectional data flow. Keeping this in mind, we can deduce that if a prop inside of our component isn't what we expect it to be, the parent component must be the culprit.

Stateful Components

Thus far, we've only talked about props, and used them in functional—or pure—components. In practice, most components shouldn't really care too much about storing state. They get some new data, rerender what needs to change as their props change, and go about their lives carefree. Pure components are easy to test, debug, and they tend to be more reusable (even across applications) because they usually don't depend on how the rest of the application works.

So far, we've been using stateless components to create static, unchanging UIs. In a real application, we'd want to give some of our components more complex, dynamic behavior. I mentioned briefly that writing ES6 classes was going to be absolutely vital for your React knowledge: you need classes for stateful components. If we're being technical, ES6 classes are just syntactic sugar for JavaScript constructor functions. So...

What is "State"?

Basically, state is a special type of property attached to a class component that can contain data we want to associate with that component. Values stored in a component's state are different from regular variables because unlike regular variables, when a component updates its state, the React application will update itself in the browser to reflect the change wherever necessary.

What's the Difference?

A component can set and update its own state, whereas its props are always received from up above and considered immutable (can't/shouldn't be changed). Additionally, the state should be set directly (state = {...}) only once in the constructor, otherwise this.setState(...) should be used. The latter method tells React to re-render the component where necessary.

Let's see an example of this in action with a simple click counter (14-Ins_BasicState).

Class components need a render() method and normally use a constructor. Constructors aren't strictly necessary, but you'll be using them more often than not. There are tons of different built-in functions that React gives us, like componentDidMount, componentWillReceiveProps, etc. These are all easily referenced in the API docs.

Overall:

  • We use state to associate data with a component and keep track of any values we want to update the UI when changed.
  • We can define methods on a class component and pass them as props.
  • The onClick prop can be used to set a click event listener to an element.
