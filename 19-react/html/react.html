What is React?

Simply put, React is a front-end JavaScript framework that focuses on interactivity and dynamic content whose engine minimizes the amount of re-rendering of components when that content changes. It's loosely object-oriented and focuses on breaking a front-end down into components—each component has its own properties and state and is completely standalone. Still with me?

Components

A component is a self-contained module responsible for producing part of an application's UI and functionality. This is radically ifferent from the concept of separating the view and logic layers. When we split up our application into components, we're still separating its concerns, but we aren't necessarily separating its technologies.

Example: with Express, Mongo and Handlebars, you provide data to a Handlebars template that renders a chat client. In React, you have separate components for the chat view, reaction button, share menu, etc. that all have their own logic and take different slices of data.

This might seem weird at first, but today's class is just a primer, not comprehensive coverage. Let's do some basic installation and documentation research.

Create React App

Now that everyone has create-react-app installed, let's talk about it and use it. It should be noted that this package was not always available: the old way of React installation involved a tedious process involving Webpack. Speaking from experience, it was pretty terrible! create-react-app is as easy as running the following:

create-react-app reactpractice

What this command does is create the scaffolding for a totally new React application, complete with the name we specify. How about running it? 

yarn start

And viewing it is as easy as going to localhost:3000. By default, this command starts our app in development mode, i.e. this "mode" is meant for development as our application will auto-reload as we make changes to it. Pretty convenient!

The SRC Folder

The generated boilerplate is interesting: we've got an src folder automatically processed by Babel, then bundled into a single bundle.js file by Webpack, which is the server React applications run on. You can see what the combined bundle.js file looks like if you were to navigate to localhost:3000/static/js/bundle.js in your browser. This folder is where our React components will live. These files normally produce JSX, which corresponds to the web page we have rendered in the browser.

We have an index.js file in the src directory. This is where execution of our React application begins. In this way, it is similar to the server.js file we've been starting all of our node applications with. In order to run any code in our React app, it needs to be directly written here, or else required/imported here.

We can also put any tests we'd like to associate with our components. Create React App ships with Jest), a library for running JavaScript tests that work with React out of the box. For our purposes of learning to use React to build applications, we won't dive into writing tests for React apps, but it's nice to know that we can without any extra overhead. (General note: Jest is awesome. Syntax is nearly identical to Mocha and Chai. Highly recommended.)

The Public Folder

This one should be fairly self-explanatory based on your static Express usage. This folder is primarily for containing the index.html that will eventually be served to users visiting our application. The #root element contains our entire rendered, React application, much like the :root pseudoelement contains all of our static HTML. When we build our React application for production, a bundle.js file containing all of our application's JavaScript is generated and added inside of the index.html file.

We can also store files in this folder which we don't want to be processed by Webpack and Babel. For example, we can place static CSS files here or link to external CDNs in the head tag of the index.html. This particular use case is probably the most familiar.

We're going to see this setup over and over again, and there isn't a need to completely memorize every aspect of the Create React App boilerplate right now. The most important takeaways are:

  1. Most code goes in src
  2. The entry point into a React app is the index.js
  3. We start up a React app in development mode with yarn start, where it live-updates. This is why we run it on a server, despite being served client-side to the user.

CSS Frameworks and React

There exist plenty of CSS frameworks, as you're already aware. Materialize, Bootstrap, Semantic UI, you name it—they're all compatible with React. Of course, there is a key difference: React uses className instead of class. Why?

Recall that in ES6, class is a reserved word. Since this word denotes a new class declaration, React uses className instead. There are some other differences between HTML and JSX we'll cover soon. As a reminder, since JSX isn't HTML, there are some caveats. JSX is syntactic sugar for calls to the React.createElement method, and according to Facebook's React Docs: "JSX is closer to JavaScript than HTML".

Some key points: We can write comments inside of our JSX using JavaScript's multi-line comment syntax inside of curly braces. JSX Variables Review

Let's go over the solved version of this activity.

Much like how className has different syntax from HTML, CSS looks different as well. We can import CSS directly into our components, which is really convenient. In doing so, we aren't saving the stylesheet to a variable. Instead, whenever we import a stylesheet, an internal style tag is created and added to the document.

This approach may be a good idea if we want to keep our component more closely coupled to its styles. How many times have you worked on a project or homework assignment and had a CSS issue because of CSS that was a little messy? By separating our CSS files by component, we'll generally have smaller, more manageable stylesheets. Adding new styles or changing existing ones should be easier as our applications grow larger.

Though it is not recommended, we are also able to add inline styles to JSX elements. We can set a style similarly to how we can when writing regular HTML code, but with two key differences:

  1. React expects the style property to be an object, rather than a string.
  2. We're using JSX curly braces to embed the style object.

Object properties are camelCased and non-numeric values are strings. Numeric values will default to pixels if a unit of measurement is expected. It is recommended that you conditionally render classes instead of inline styles. Of course, it's up to you to decide how we want to organize your component styles, but there is value in being consistent in whichever approach or hybrid approach you take.
