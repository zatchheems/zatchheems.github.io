CSS Purpose, Syntax, and Styles

HTML is the content, CSS makes it pretty.

selector { property : value ; }

We can add styles to elements by referencing them directly, adding classes, referencing them by ID. We can also reference their children with >.

Editable CSS properties: margin, padding, border, width, height, color, background-color, font, font-size...

Floating

Floating an element removes it from the regular flow of content. Elements can be floated left or right, or clear can be specified to ignore floated elements.

Floats are weird! Use with caution.

display:

  • inline
  • block

Positioning

static, relative, absolute, and fixed.

static is the default positioning.

relative sets an element for absolute elements to be positioned against.

fixed fixes an element to the viewport.

Box Model

Total width = margin-left + border-left + padding-left + width + padding-right + border-right + margin-right

Total height = margin-top + border-top + padding-top + width + padding-bottom + border-bottom + margin-bottom

The Double Take

Why use divs vs. sections and other HTML5 semantic tags (header, nav, section, aside, article, details, summary, footer)? It turns out neither is more "correct" per se. Most sites still stick to just divs. Semantic tags keep things clean and organized and help with search engine optimization, but regular divs can be styled semantically with classes/IDs.

CSS Resets

Loading multiple CSS files simultaneously:

<link rel='stylesheet' href='assets/main.css'>
<link rel='stylesheet' href='assets/extra.css'>
<link rel='stylesheet' href='assets/even-more.css'>

We can use multiple stylesheets to create extremely sophisticated styling for our sites and have fine-grained control over how our content is styled. What we get is a totally unique page that doesn't need to change the main CSS for the other pages. It's a great trick!

This HTML document uses the previous notes' CSS file plus another that overrides some of the first while adding some more stuff. We can demonstrate that by just commenting out the new <link ...href='new-styles.css'> tag in the head.

Loading order matters!

CSS Resets for Browser Compatibility

Quick vote: what browser do you use the most?

Each browser has a different rendering engine under the hood. Things will often look different between Chrome, Firefox, Safari, Edge, Opera, etc. We can combat this with a reset.css stylesheet. Basically, what this does is ensures that your page renders identically between all browsers by overriding all browser-specific default CSS.

The main caveat? Everything has to be re-styled from there. (It's a good trade-off!)

Deployment

What does this mean, exactly?

Right now, your HTML is available only on your local machine. This is just fine for testing purposes, but how will others be able to access it?

We have to actually upload it to a server whwere it can be hosted before we can show it off elsewhere. A host in this case is a web server capable of hosting our content and delivering it when given a properly-formed HTTP request it understandsâ€”when you navigate to a URL, your browser sends HTTP requests to download content and render it in the viewport (amongst a wealth of other things that happen under the hood).

Instructor activity...

We've been concerned just with the rendered part, so lets tackle hosting.

Let's use GitHub Pages to serve some content. The idea is that we can turn a GitHub repo into a live webpage without having to upload our code anywhere else. The steps to do so are relatively simple:

  1. Create a new repo, github-username.github.io
  2. Navigate into a folder and clone the repo into it
  3. Add an HTML file named index.html containing the code for your webpage
  4. Add, commit, and push your changes into the repository
  5. Navigate your browser to github-username.github.io and we're live!
