Python and virtualenv Setup

    
Before we begin, we will verify that our environments are identical—barring operating system differences, of course—such that
      we don't have any version-specific problems or errors. The same rings true for Node and NPM versions: different versions will actually function
      differently. It is common to use NVM for Node.

    
However, since we're in Python land, we have some different tools to manage, specifically Python itself. Python 3.x is the version we want. Many
    older machines come with 2.7.x by default. This difference in numbering is known as a major revision, and as such, breaks a
    lot of old stuff and adds a lot of new stuff. Run the following to check that you have Python 3 and above:

    
python --version

    
If you're rocking above 3.0, you're in good hands. You can immediately run the following to initialize a new virtual environment, which will
    keep a totally standalone, local instance of whatever is inside it (just like installing Node packages locally):

    
# replace "env" here with whatever you like.
ENV="env"
virtualenv $ENV

    
For those with Python versions less than 3, your command will look like this:

    
# env can be whatever you like, but should be the same as above
virtualenv -p `which python3` env

    
You can find the path to your Python3 install by running which python3. You can use inline
    command substitution to do this with backticks ("`"), like above.

    
To activate the virtualenv, run:

    
source $ENV_NAME/Scripts/activate

    
Easy as that, hopefully. We'll help troubleshoot any rockiness along the way.

    

    

      
Django and the MVC

      
As mentioned briefly before, Django is an MVC framework. Why is this important?
        Well, say you're building a Node app with Express. You have complete control over 
        exactly how you'd like your data to be saved and structured.
        This is powerful, but super inconvenient, as you are now required to "glue" everything
        together manually.
      

      
This brings into question the advantages of convention of configuration.
        It's like the ease of use of React vs. the lightness and customizability of Handlebars.
        One requires a lot more overhead; the other requires more code.
      

      
Django favors convention. It follows a more MTC structure,
        using templates. Django's "views" have controller-like responsibilities. Think about 
        how Express can choose to send raw HTML. Upgrade that to accept Handlebars-like templates
        and we're good to go.

      
Django Installation and Create Project

      
Time to actually install this thing! Be sure to source your virtualenv if you haven't
        already. After doing so, use Python's package manager to install Django like so:
      

      
# note: you don't have to use pip3 within a virtualenv
pip install Django==1.10.1

      
Django provides a command-line tool for creating new projects called django-admin.
      Create a new project with the following:

      
# feel free to name it whatever you like.
# I'll be using "app"
django-admin startproject app

      
If you run tree ., you'll see the entirety of the app's contents.

      
The app directory contains project-wide configuration filesâ€”this is where we set up database configuration, tell Django
      where static files are located, etc.

      
manage.py is where Django's utility toolchain lives.
      This is the tool you'll use to migrate databases, run the development server, create administrators, etc.

      
This directory contains all of the files for a Django project, but we develop specific functionality in what
      Django calls apps.
      Ideally, each app is responsible for a specific set of related features.
      A web app is a collection of different "feature sets." A Django project is the web app; each "feature set" is a Django
      app.

      
To initialize a new app, we use the manage.py tool.
      Create a new app with python manage.py startapp users.
      The general formula is pythonmanage.py startapp $APP_NAME.

      
Running this command creates a new folder with our app name. This directory is where we'll spend
        most of our development time.
      

      
Finally, when you create a new app like this, you need to "register" it in Django's internal settings
      file. This allows it to find important, app-specific files. Within app/settings.py,
      under INSTALLED_APPS, we'll add users.apps.UsersConfig
      to set up this new app called "users". This is a one-time thing we do for each new app.

    

    

      
Views and URLs

      
Since Django follows the MVC paradigm, we'll be exploring the Controller layer
        first, since this is what allows us to bind UIs to data. As a reminder, the 
        "controller" in MVC is called a "view" in Django. Let's check out users/views.py.
        This is where things might get confusing. You have to remember that views.py
        is where you define controllers, not UI details.
      

      
Each view is just a Python function. When a user hits a URL/route on your website,
        Django calls a function defined in views.py, much like Express.
        It passes the request as a parameter to each view function—here's where you determine
      if it was a GET or POST. You need to tell Django which function to call when a user hits a given route elsewhere. We'll get to that after
      defining a few view functions.
